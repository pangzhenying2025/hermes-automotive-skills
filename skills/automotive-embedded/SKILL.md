### Cross-Compilation and Embedded Build Skills

## Embedded Cross-Compilation Guide

### Overview

Cross-compilation builds executable code on a host machine (x86_64 Linux/Windows)
for a different target architecture (ARM Cortex-M, Cortex-A, RISC-V). This is
essential for automotive embedded development where ECU hardware has limited
resources and cannot host a compiler.

### Toolchain Architecture

```
Host (x86_64 Linux)                    Target (ARM Cortex-M4)
┌──────────────────────┐              ┌──────────────────────┐
│ arm-none-eabi-gcc    │  ──────►     │ STM32F4 MCU          │
│ arm-none-eabi-ld     │  .elf/.hex   │ 1MB Flash            │
│ arm-none-eabi-objcopy│              │ 192KB RAM             │
│ arm-none-eabi-gdb    │  ◄──────     │ OpenOCD/ST-Link      │
└──────────────────────┘   debug      └──────────────────────┘
```

### Common Automotive Toolchains

| Toolchain | Target | Usage |
|-----------|--------|-------|
| arm-none-eabi-gcc | Cortex-M (bare-metal) | MCU firmware (BMS, gateway) |
| arm-linux-gnueabihf-gcc | Cortex-A (Linux) | Linux-based ECUs |
| aarch64-linux-gnu-gcc | Cortex-A53/A72 (Linux) | HPC, ADAS platforms |
| qcc (QNX) | Various ARM/x86 | QNX RTOS targets |
| tricore-gcc | Infineon TriCore | Engine/transmission ECUs |

### Compiler Flags for ARM Cortex-M

```makefile
# Target CPU and instruction set
MCPU = -mcpu=cortex-m4
MTHUMB = -mthumb

# Floating point
MFPU = -mfpu=fpv4-sp-d16
MFLOAT = -mfloat-abi=hard    # hard = HW FPU, soft = SW emulation

# Optimization
OPT = -Os                     # Size optimization for flash-limited MCUs
# OPT = -O2                   # Speed optimization for performance-critical

# Code generation
CFLAGS = $(MCPU) $(MTHUMB) $(MFPU) $(MFLOAT) $(OPT)
CFLAGS += -ffunction-sections -fdata-sections  # Enable garbage collection
CFLAGS += -Wall -Wextra -Werror               # Strict warnings
CFLAGS += -std=c11                             # C11 standard

# Linker flags
LDFLAGS = -T linker_script.ld
LDFLAGS += -Wl,--gc-sections    # Remove unused sections
LDFLAGS += -Wl,-Map=output.map  # Generate memory map
LDFLAGS += --specs=nosys.specs  # No system calls (bare-metal)
```

### Linker Script Essentials

```ld
MEMORY {
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 1024K
    RAM (rwx)   : ORIGIN = 0x20000000, LENGTH = 192K
    CCMRAM (rw) : ORIGIN = 0x10000000, LENGTH = 64K
}

SECTIONS {
    .isr_vector : { KEEP(*(.isr_vector)) } > FLASH
    .text       : { *(.text*) } > FLASH
    .rodata     : { *(.rodata*) } > FLASH
    .data       : { *(.data*) } > RAM AT > FLASH  /* LMA in Flash, VMA in RAM */
    .bss (NOLOAD) : { *(.bss*) *(COMMON) } > RAM
    ._stack (NOLOAD) : { . = ALIGN(8); . += 8K; _estack = .; } > RAM
}
```

### Startup Code Requirements

The startup code (usually assembly) must:
1. Set stack pointer to top of RAM (_estack from linker script)
2. Copy .data section from Flash (LMA) to RAM (VMA)
3. Zero-initialize .bss section
4. Enable FPU if target has one (write to CPACR register)
5. Call SystemInit() for clock configuration
6. Call __libc_init_array() for C++ static constructors
7. Call main()

### Binary Output Formats

```bash
# ELF to raw binary (for flash programming)
arm-none-eabi-objcopy -O binary firmware.elf firmware.bin

# ELF to Intel HEX
arm-none-eabi-objcopy -O ihex firmware.elf firmware.hex

# ELF to Motorola S-Record
arm-none-eabi-objcopy -O srec firmware.elf firmware.s19

# Check binary size
arm-none-eabi-size firmware.elf
#   text    data     bss     dec     hex filename
#  45232    1024    8192   54448    d4b0 firmware.elf
```

### Memory Usage Analysis

```bash
# Generate map file (add -Wl,-Map=output.map to LDFLAGS)
# Map file shows section sizes, symbol addresses, and memory regions

# Find largest functions
arm-none-eabi-nm --size-sort firmware.elf | tail -20

# Check section sizes
arm-none-eabi-size -A firmware.elf
```

### RTOS Integration

When porting FreeRTOS or Zephyr to the target:
- Configure system tick timer (SysTick for Cortex-M)
- Set heap size in FreeRTOSConfig.h or linker script
- Configure interrupt priorities (PendSV, SVCall for context switch)
- Reserve stack space per task (minimum 256 bytes, typical 1-4KB)

### Debugging

```bash
# Start OpenOCD for STM32
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg

# Connect GDB
arm-none-eabi-gdb firmware.elf
(gdb) target remote :3333
(gdb) monitor reset halt
(gdb) load
(gdb) break main
(gdb) continue
```

### Best Practices

- Pin toolchain version in CI/CD for reproducible builds
- Use -Os for flash-constrained MCUs, -O2 for CPU-constrained
- Enable -ffunction-sections + --gc-sections to remove dead code
- Always generate and review the map file for memory usage
- Test on host with native build (mocks for HAL) before target
- Use static analysis (cppcheck, MISRA checkers) in build pipeline
- Keep bootloader and application in separate build targets
