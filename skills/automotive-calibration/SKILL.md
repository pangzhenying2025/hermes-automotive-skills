### ECU Calibration Skills

## ECU Calibration Guide (XCP/INCA/CANape)

### Overview

ECU calibration is the process of tuning software parameters in an electronic control
unit to achieve desired vehicle behavior. Parameters such as fuel maps, torque curves,
PID gains, and threshold values are adjusted using measurement and calibration (M&C)
tools connected via the XCP (Universal Measurement and Calibration Protocol) protocol.

### XCP Protocol Architecture

```
Calibration Tool (Master)          ECU (Slave)
┌──────────────────┐              ┌──────────────────┐
│ INCA / CANape    │  XCP on ETH  │  Application SW  │
│                  │◄────────────►│  + XCP Slave     │
│ A2L Database     │  XCP on CAN  │                  │
│ Parameter Editor │  XCP on USB  │  Flash / RAM     │
└──────────────────┘              └──────────────────┘
```

XCP operates in a command-response model where the master sends commands and the
slave responds. Key XCP commands include:

| Command | Code | Purpose |
|---------|------|---------|
| CONNECT | 0xFF | Establish connection |
| DISCONNECT | 0xFE | Close connection |
| SET_MTA | 0xF6 | Set memory transfer address |
| UPLOAD | 0xF5 | Read data from slave |
| DOWNLOAD | 0xF0 | Write data to slave |
| GET_DAQ_RESOLUTION_INFO | 0xD9 | Query DAQ capabilities |
| ALLOC_DAQ | 0xD5 | Allocate DAQ lists |
| START_STOP_DAQ_LIST | 0xDE | Control measurement |

### A2L Database Structure

The ASAP2 (A2L) file is the central description database linking parameter names
to ECU memory addresses. Key A2L objects:

- **MEASUREMENT**: Read-only signals (RPM, temperature, pressure)
- **CHARACTERISTIC**: Tunable parameters (maps, curves, scalars)
- **COMPU_METHOD**: Conversion formulas (raw to physical values)
- **RECORD_LAYOUT**: Memory structure description
- **MOD_PAR**: Module parameters (EPK, memory segments)

A2L parsing example (physical value conversion):
```
physical = (raw_value * factor) + offset
Example: Temperature = (raw * 0.1) - 40.0  [degC]
```

### Calibration Workflow

1. **Connect**: Establish XCP session with target ECU
2. **Load A2L**: Parse the A2L database for address and type information
3. **Measure**: Setup DAQ lists to capture real-time signal data
4. **Calibrate**: Modify parameters in RAM using DOWNLOAD commands
5. **Validate**: Observe system response through measurements
6. **Flash**: Program validated calibration to non-volatile memory
7. **Verify**: Read back and confirm calibration integrity

### DAQ Measurement Configuration

Data Acquisition (DAQ) enables high-speed, time-synchronized measurement:

- **DAQ List**: A collection of ODT (Object Descriptor Table) entries
- **ODT Entry**: Maps a specific memory address and size for capture
- **Event Channel**: Trigger source (e.g., 10ms task, crank angle)
- **Prescaler**: Reduce capture rate (every Nth event)

Typical DAQ rates: 1ms (engine control), 10ms (chassis), 100ms (body)

### Parameter Types

- **Scalar**: Single value (e.g., idle speed setpoint = 800 RPM)
- **Curve**: 1D lookup table (e.g., boost pressure vs. RPM)
- **Map**: 2D lookup table (e.g., fuel injection vs. RPM and load)
- **Value Block**: Array of related values
- **ASCII**: String parameters (e.g., calibration version ID)

### Calibration Data Formats

| Format | Extension | Usage |
|--------|-----------|-------|
| Intel HEX | .hex | Flash programming |
| Motorola S-Record | .s19 | Flash programming |
| DCM | .dcm | ETAS calibration exchange |
| CDF | .cdf | CANape calibration data |
| PaCo | .paco | Parameter container (AUTOSAR) |

### Safety Considerations

- Always validate parameter values against min/max limits from A2L
- Use write protection for safety-critical parameters (ASIL-rated)
- Implement calibration access levels (read-only, limited write, full write)
- Log all calibration changes with timestamp, user, old/new values
- Perform regression testing after calibration changes
- Keep backup of original calibration before modifications

### Best Practices

- Use working page (RAM) for online tuning, reference page (Flash) as baseline
- Group related parameters in calibration datasets for version control
- Use parameter sweep automation for systematic optimization
- Document calibration rationale (why a value was chosen, not just what)
- Compare calibration datasets between SW versions to detect unintended changes
- Validate measurement system accuracy (sensor scaling, sample rate) before tuning
- Use XCP on Ethernet for high-bandwidth DAQ; CAN for simple parameter access
